# Формулировка проблемы в терминах задачи МО

## Определение цели МО

Рассмотрим три возможные цели МО:
- максимизировать метрику `Recall@160` через ранжирование на основе взвешенной суммы предсказанных типов контактов;
- максимизировать метрику `Recall@160` через "жадное" ранжирование по глобальной вероятности любого целевого контакта;
- максимизировать метрику `Recall@160` через ранжирование по вероятности контакта со встроенным системным балансом по вертикалям.

Обсудим каждый вариант подробнее.

### Вариант 1. Ранжирование на основе взвешенной суммы реакций

В этом случае система обучается предсказывать каждый из 11 типов целевых контактов отдельно. Итоговый топ-160 формируется путем сортировки кандидатов по взвешенной сумме этих вероятностей, где каждому типу назначен свой вес.

Преимущество этого варианта в том, что он потенциально позволяет отличить "сильные" сигналы (например, звонок продавцу) от "слабых" (открытие чата) и поднять их выше в выдаче.
К недостаткам относится то, что идентификаторы действий `eid` полностью анонимизированы. У нас нет объективных бизнес-оснований назначать одному событию вес больше, чем другому. Кроме того, целевая метрика соревнования `Recall@160` бинарна: она просто считает факт пересечения предсказания с реальностью и не дает дополнительных баллов за предсказание конкретного типа контакта.

### Вариант 2. "Жадное" ранжирование по вероятности контакта (глобальный CVR)

Здесь все 11 контактных событий интерпретируются как равнозначные. Модель обучается предсказывать вероятность наступления хотя бы одного из целевых событий, и система просто отбирает 160 кандидатов с наивысшим *скором* по всему каталогу.

Преимущество подхода заключается в прямой и простой оптимизации под механику целевой метрики соревнования.
Главный недостаток в том, что "жадная" сортировка заполнит рейтинг товарами из одной-двух самых конверсионных вертикалей (например, дешевыми товарами) и проигнорирует остальные. В условиях жесткого бакетинга пользователей по категориям `(v0, v2, v3, v4, v57)` это неизбежно приведет к обвалу метрики в "нелюбимых" моделью корзинах.

### Вариант 3. Ранжирование с учетом баланса — алгоритмической справедливостью по вертикалям

В этом случае цель МО — отобрать максимум релевантных объявлений, но с внедрением строгого системного ограничения: качество выдачи не должно зависеть от принадлежности объявления к конкретной вертикали `vertical_id`. Максимизация контактов не должна происходить за счет игнорирования нишевых категорий. Баланс обеспечивается на архитектурном уровне системы, а не путем усложнения функции потерь базового классификатора.

### Какой вариант выбрать?

Мы выбираем **третий вариант**, поскольку он напрямую решает бизнес-задачу соревнования. Прямая максимизация вероятности контакта соответствует механике расчета Recall@160, а архитектурное требование баланса страхует систему от провала на нишевых вертикальных бакетах пользователей.

## Определение входных и выходных данных системы

Глобально система принимает на вход профиль пользователя, контекст его исторического поведения и каталог доступных объявлений. На выходе формируется отсортированный список ровно из 160 релевантных `item_id`.

Поскольку оценивать весь каталог тяжелой моделью вычислительно невозможно, логично было бы рассмотреть архитектуру с многоступенчатым отсеиванием, и, ввиду характеристик постановки задач, было принято решение остановиться на классической двухэтапной рекомендательной системе. 

### Шаг 1. Этап генерации кандидатов (Retrieval)

На этом этапе из всего каталога отбирается компактный пул кандидатов с максимальным охватом. (Новые объявления без истории — предметы холодного старта — не отфильтровываются, а можно подмешиваются в пул отдельной веткой на основе их контентного сходства, чтобы обеспечить выдачу нового контента.)
#### Входные данные
- Анонимизированный идентификатор пользователя `user_id`.
- Исторический кликстрим пользователя (действия до отсечки `threshold_date`).
- Атрибуты объявлений `item_features`: категория, цена, гео, текстовые метаданные.
#### Выходные данные:
- Фиксированный пул из $M$ ~ $10^3–10^4$ кандидатов-объявлений для каждого пользователя, сформированный ансамблем алгоритмов.

### Шаг 1.5. Бизнес-логика 1 (Hard Filters)

Полученный пул кандидатов обязательно проходит через жесткие фильтры перед передачей в ранжировщик
- **Проверка доступности:** Исключение закрытых, забаненных или неактивных на момент предсказания объявлений.
- **Отсев просмотренного (`anti-join`):** Исключение объявлений, с которыми пользователь уже контактировал.
- **Решение проблемы Over-fetching:** Чтобы пул не истощился после `anti-join`, ретривер изначально запрашивает кандидатов с запасом ($M \gg 160$). Если пользователь аномально активен и после вычитания истории в пуле остается мало айтемов, применяется стратегия **Fallback** — недостающий объем пула "добивается" популярными объявлениями, будь то по региону или любимым категориям пользователя. 

### Шаг 2. Этап точного ранжирования (Ranking)

На этом этапе происходит тяжелая оценка вероятности целевого действия для отфильтрованного пула кандидатов.
#### Входные данные
- Обособленные пары $\langle User, Item \rangle$ из отфильтрованного пула кандидатов. Для каждой пары формируется вектор признаков, включающий:  
- Профиль пользователя: агрегаты исторической активности и распределение интересов по вертикалям.
- Атрибуты объявления: контентные метаданные (категория, цена), гео-признаки и семантические эмбеддинги (`sid_0...sid_3`).
- Фичи кросс-пересечения (например, близость эмбеддинга пользователя и объявления).

#### Выходные данные
- Предсказанная вероятность наступления целевого события — $P(\text{Contact}=1 \mid \text{User}, \text{Item})$ — для каждого кандидата из пула. Полученное непрерывное значение $\hat{y} \in [0, 1]$ напрямую используется в качестве скора для итоговой сортировки 160 лучших `item_id`.

### Шаг 2.5. Бизнес-логика №2 (Reranking & Balancing) 

- Берем условные топ-500 со скорами, применяем логику "баланса по вертикалям" и спускаемся сверху вниз, пока не наберем **ровно 160** лучших и сбалансированных постов.
- Результат сериализуется в требуемый формат и сохраняется в итоговый сабмит-файл.

## Выбор категории МО

Система рекомендаций должна сформировать персональный список из 160 объявлений, отсортированных по вероятности целевого контакта. Глобально эту задачу можно отнести к **задачам ранжирования (Learning to Rank, LTR)**. 

Цель системы — отсортировать объекты так, чтобы максимизировать вероятность целевого действия на самом верху списка, сохранив алгоритмическую справедливость.
Однако из-за огромного пространства поиска (миллионы объявлений и 5 миллиардов исторических событий) применить единую модель ранжирования ко всему каталогу практически невозможно. Поэтому концептуально решение задачи разбивается на 2 логические части, использующие разные парадигмы МО:

### 1. Самообучение и обучение без учителя (Self-Supervised / Unsupervised Learning)

- **Цель**: Масштабное сужение пространства поиска и выявление скрытых взаимосвязей.
- **Результат**: Формирование компактного пула релевантных кандидатов для каждого пользователя.

### 2. Обучение с учителем (Supervised Learning)

- **Цель:** Точное предсказание вероятности целевого контакта для ограниченного пула кандидатов.
- **Результат:** Каждому кандидату присваивается точный скор релевантности.
